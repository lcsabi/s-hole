# CL 29 ‚Äî Hardening batch: goleak, govulncheck, empty-blocklist alarm

## Description

A batch of three low-level robustness improvements (roadmap item #6):

1. **Goroutine-leak detection** ‚Äî `go.uber.org/goleak` runs the test suites of
   the three goroutine-heavy packages (`cache`, `querylog`, `dnsserver`) under
   `TestMain`, failing the package if any goroutine outlives the tests.
2. **Vulnerability scanning** ‚Äî `govulncheck` runs as its own CI job and a
   `make vuln` target, failing on any known CVE reachable from our code.
3. **Empty-blocklist alarm** ‚Äî `blocklist.Update` now emits a loud `WARN`
   whenever the block set ends up empty, so an operator notices s-hole is
   answering queries while blocking nothing.

## Motivation

The roadmap originally scoped a third item here ‚Äî an **embedded fallback
blocklist** (`//go:embed` ~1000 core ad domains) so a first run with no
network still filters. We dropped it in favor of the alarm. Reasoning:

- The scenario it targets is vanishingly narrow. s-hole's core job ‚Äî forwarding
  queries upstream ‚Äî already requires network; a box that cannot reach a
  blocklist URL usually cannot reach `1.1.1.1:53` either. The on-disk cache
  already covers every restart after one successful download. The fallback only
  helps on a *first* run where upstream DNS works but blocklist HTTP downloads
  fail and no cache exists yet.
- A vendored list is stale the day it is committed, carries licensing and
  provenance baggage, adds a doc-sync surface, and bloats the binary ‚Äî all of
  which cut against the "auditable in an afternoon" identity.
- It *masks* the real problem (nothing loaded) instead of surfacing it.

The alarm is the better hardening: it makes the failure loud rather than
papering over it. It also closes a real gap ‚Äî a source that returned HTTP 200
but parsed to zero domains previously logged `blocklist updated total=0` at
**Info**, indistinguishable from a healthy refresh.

goleak and govulncheck are standing safety nets: the first pins the
concurrency invariants the shutdown path depends on (the cache cleanup
goroutine, the DBLogger writer/prune goroutines, the DNS listeners must all be
reclaimed on Close/Shutdown); the second turns a new CVE advisory into a
red CI check rather than something noticed by chance.

## Design decisions

- **goleak scope**: only `cache`, `querylog`, `dnsserver` ‚Äî the packages that
  own long-lived goroutines. `blocklist` is deliberately excluded: its tests
  use `httptest`, whose keep-alive connections can register as transient
  goroutines and produce flaky leak reports. All three target packages already
  pass clean ‚Äî no leaks were found, the guard is preventive.
- **govulncheck delivery**: the official `golang/govulncheck-action` as a
  standalone CI job (independent of build/test so a new advisory surfaces on
  its own), plus a `make vuln` target that uses `go run ‚Ä¶/govulncheck@latest`
  so the tool never enters the module's `require` set.
- **Alarm placement**: inside `blocklist.Update` (via `warnIfEmpty`) rather than
  in `cmd/s-hole/main.go`, so it covers both call sites ‚Äî the initial load and
  the periodic/`POST /api/reload` refresh ‚Äî and both exit paths (all-sources-
  failed and parsed-to-empty). WARN, not ERROR: it is a state warning, distinct
  from the existing "all sources failed" ERROR which flags a transient fetch
  problem.

  cover both sides of the alarm; `blocklist` coverage 89.6% ‚Üí 89.9%.
- All three goleak-guarded packages pass under both the plain and `-race`
  runners; no goroutine leaks surfaced.
