# CL 25 — dashboard: Cache Hit Rate stat card

**Bug:** —

## Description

The `/api/stats` payload has carried `cache_hits` and `cache_hit_pct`
since the cache landed (CL 5), and both the README and DESIGN route
tables describe the endpoint as returning the cache hit rate — but the
dashboard never rendered it. The cards row showed Total Queries /
Blocked / Block Rate only, so the only ways to see the hit rate were
`/metrics` or the periodic stats log line.

The hit rate is an operational number, not a vanity one: it is the
signal for whether `cache_size` is tuned to the network (the README
quotes typical rates of 40–70 %), and it directly explains upstream
query volume. This CL adds a fourth stat card, **Cache Hit Rate**,
bound to the `cache_hit_pct` field the UI already downloads on every
3-second poll.

Pure static-UI change inside the existing design: the cards grid goes
from `repeat(3, 1fr)` to `repeat(4, 1fr)`, one card `<div>` is added,
and one line in `refresh()` writes the value. No API, config, or
handler changes; explicitly *not* the "Web UI redesign" non-goal.

## Testing

`go vet` + `go test ./internal/api/` pass (the UI is a `go:embed`
asset; no Go behaviour changed). Verified live on the VM deployment
(192.168.100.18, the CL-24-era rehearsal machine): rebuilt the amd64
binary, `install` + `systemctl restart`, confirmed the served HTML
contains the new card, drove two identical A queries through the
sinkhole, and watched `/api/stats` report `cache_hit_pct: 50` with the
dashboard card showing **50.0%** on the next poll.
