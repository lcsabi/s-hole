# CL 24 — make the lint gate real: golangci-lint v2 compliance

**Bug:** —

## Description

The first CI run on GitHub failed in the lint job — the first time
golangci-lint ever actually ran against this repo (it was not installed
on the development machine; `make check`'s lint step had been silently
absent). Two root causes:

1. **golangci-lint v2 removed the v1-era default exclusions** (the old
   `EXC0001` preset) that silenced idiomatic errcheck patterns:
   `fmt.Fprint*` to writers, deferred `Close` on read paths,
   best-effort `os.Remove`, `tx.Rollback` in error paths. Our config
   declares `version: "2"`, so the v2 binary CI installed applied the
   stricter defaults and reported 43 errcheck findings plus one
   staticcheck QF1003.
2. **`make tools-install` installed the wrong major version**: the
   un-versioned module path (`github.com/golangci/golangci-lint/cmd/…`)
   resolves to v1, which cannot parse a `version: "2"` config at all —
   so even a developer who ran `make tools-install && make lint` would
   have hit a config error, not the findings.

Resolution mixes config, code improvements, and explicit ignores:

- **`.golangci.yml`**: `errcheck.exclude-functions` restores the
  idiomatic subset (`fmt.Fprint/Fprintf/Fprintln`, `(*os.File).Close`,
  `(io.ReadCloser).Close`, `(*database/sql.Rows).Close`,
  `(*database/sql.Stmt).Close`) with a comment stating the contract:
  everything excluded is a best-effort call where no data can be lost;
  writes that CAN lose data (`DBLogger.Close`, `FileLogger.Close`,
  cache-file renames) keep checking their errors in code.
- **`dnsserver.Server.Shutdown`** now logs listener shutdown errors
  instead of discarding them — a genuine b/013-class improvement.
- **Explicit `_ =` with rationale comments** for true best-effort
  calls: the three `.tmp` cleanup removes in `fetchList`, the two
  `db.Close()` calls in `NewDBLogger`'s error paths, and
  `tx.Rollback()` in `flush` (in each case the earlier error is the
  actionable one).
- **`FileLogger.Log`** doc comment now states the best-effort contract
  (same philosophy as DBLogger's drop-on-full channel).
- **`api_test.go`**: if/else-if chain on `out["status"]` converted to
  a tagged switch (staticcheck QF1003).
- **Makefile `tools-install`** installs from the `/v2` module path.

## Addendum (same day)

The CI log showed the actual CI failure was upstream of any finding:
`golangci-lint-action@v6` only installs golangci-lint **v1** (v1.64.8,
built with Go 1.24), which refuses to load a config targeting Go 1.25
— CI never linted a single file. The original diagnosis ("CI reported
these findings") was therefore wrong; the findings were reproduced
locally with the real v2 binary and were a genuine latent failure that
would have surfaced the moment the action was fixed. Follow-up commit
bumps the action to `golangci-lint-action@v8`, which installs
golangci-lint v2. Both halves are required for a green lint job.
