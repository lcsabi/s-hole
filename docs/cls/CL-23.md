# CL 23 — dual-stack listener default + IPv6 network docs

**Bug:** —

## Description

s-hole's IPv6 story had three layers with different answers: AAAA
records were fully supported (including `::` sinkhole replies), but the
default `listen: "0.0.0.0:53"` bound an IPv4-only socket — clients
querying over IPv6 transport were silently ignored — and the README
said nothing about the dual-stack deployment trap.

**Code**

- Default `listen` changes from `"0.0.0.0:53"` to `":53"`, which binds
  a dual-stack wildcard socket (IPv4 + IPv6) on every mainstream OS.
  IPv4-only hosts are unaffected; operators who want IPv4-only binding
  set `listen: "0.0.0.0:53"` explicitly. Verified empirically: with the
  new default, the same process answers queries on `127.0.0.1` and
  `::1`.
- `TestLoad_EmptyAppliesDefaults` updated to pin the new default.

**Docs**

- `config.yaml` documents the `listen` field (dual-stack default,
  IPv4-only and single-interface alternatives).
- README config table updated; new **IPv6 networks** callout in "Point
  your router at it" documenting the RA/RDNSS bypass: dual-stack
  clients prefer the router-advertised IPv6 resolver, so unless that
  advertisement is disabled or repointed at s-hole, IPv6-capable
  devices quietly circumvent the sinkhole. (Pi-hole's docs carry the
  same warning; it is the most common "the ads came back" report on
  dual-stack LANs.)
- CONTRIBUTING smoke test uses `S_HOLE_LISTEN=:5353` to match the new
  default's style.
- CHANGELOG entry under Changed (behaviour-visible default change).
